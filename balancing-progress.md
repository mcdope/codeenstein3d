  - **Regression found + fixed — turn oscillation**: bumping `rotSpeedMultiplier` made each fixed-duration turn-key-hold cover a *bigger* angle per tick. Once that per-tick angle exceeded a target's tolerance window, the bot would overshoot past it, see the opposite-signed delta next tick, and overshoot back — forever (confirmed via trace: `dir` ping-ponging between two fixed values every tick, 50+ consecutive ticks, no convergence). Fixed with `turnBurstMs(deltaAngle, rotSpeedMultiplier)`: computes exactly how long to hold the turn key to land at (not past) the target angle, capped at the normal step for large deltas so far-off turns are just as fast as before. Applied to every turn-only decision point in `tick()` (hazard/spike escape, critical-health retreat, mine retreat, ranged/melee/nav-target turning).
  - **Deeper bug found via the same trace — melee whiffing against Edge Cases**: found a *second*, independent bug while verifying: the bot got stuck in melee range against one enemy for 400-500+ ticks straight, swinging every tick, never landing a hit (`aliveAggroed`/`hp` unchanged the whole time). Root cause: melee hit detection (`fire()` in `engine.ts`) uses the *same* screen-crosshair-column projection test as ranged weapons (`findTargetInProjections` against the exact center column) — a swing only lands within the target's on-screen width, which shrinks with distance (even inside melee range) and is smaller still for an Edge Case (`EDGE_CASE_SCALE=0.55`).   - **(a) Ranged fire without line-of-sight, wasting ammo on wall corners**: aggro is sticky (`enemyAi.ts`), so a threat that ducked behind a corner (or was chased around one) stays "aggroed" and angle-aligned even with no actual clear shot — `tick()` had no LOS check at all for ranged fire (only mines had one, via `findDisarmableMine`'s `hasLineOfSight`). Fixed by adding the same `hasLineOfSight` check before ranged firing; when blocked, the bot now advances instead (same "keep closing distance" logic as an unaligned angle) rather than standing and firing into the wall.
  - **(b) Movement "seems a bit slow, even on casual"**: root cause wasn't `MOVE_SPEED` (untouched, same 3.2 tiles/sec as always) — it was that plain navigation (no combat) never moved *while* turning, only after fully aligning (`TURN_MOVE_EPS`). Since BFS route waypoints often need at least a moderate heading correction, this meant a full stop-turn-go cycle at nearly every waypoint transition. Fixed: `tick()`'s plain nav-target branch now also walks while turning, capped at `MAX_WALK_WHILE_TURNING_RAD` (1.0 rad) so a sharp corridor doubling-back doesn't send it walking the wrong way mid-turn.
  - **(c) "A real player wouldn't walk through an acid pool, would go around it"**: confirmed on `main.c` — a real detour exists around its one hazard patch via the rooms to its north/south, but the route planner (`routePlanner.mjs`) crossed straight through it anyway. Root cause: hazard/spike avoidance was a binary all-or-nothing choice — first for the *whole route* (any single leg failing to avoid hazard discarded the entire avoiding attempt), then (a same-day first fix attempt) *per leg* — but `main.c`'s entire spawn-to-exit is one single leg/BFS call, and once crossing was permitted anywhere in that one call, plain BFS had no reason to prefer the longer detour over the shorter through-the-hazard route. Properly fixed by replacing the binary choice with a **weighted Dijkstra** (`weightedPath`, new in `routePlanner.mjs`): hazard/spike tiles cost `HAZARD_TILE_COST=25`x a normal floor tile instead of being freely walkable or fully blocked, so the path prefers a reasonably-short detour but still finds a way across when crossing is genuinely the only option. Verified on `main.c`: the route no longer touches the avoidable patch at all, while still correctly crossing the one genuinely-unavoidable spike trap elsewhere on the level. `planCoverageRoute` (still unused by any live profile) was left on the old binary approach rather than migrated, since it's dead code for now.
  - **(d) Bot skips visible loot/drops**: `maybeDetourForLoot` only ever knew about `map.ammoPickups` — static, pre-placed pickups known from Node-side map generation before any browser exists. Dynamic kill-drop loot (`this.drops` in `engine.ts`, created at runtime when an enemy dies) was never exposed to the bot at all, so it had no way to know a drop existed and would walk right past it. Fixed by adding a new `getDrops()` test hook (`{x, y, kind}[]`) and merging it into the uncollected-loot pool every `maybeDetourForLoot` call — static pickups keep the existing `visitedPickups` bookkeeping (the map's list never shrinks), dynamic drops are just re-queried live each time (the engine's own list already shrinks on collection, no separate tracking needed).

## Summary of the 5-iteration anomaly-scan loop (2026-07-14)

Real bugs found and fixed in `scripts/run-balancing-telemetry.mjs` / `scripts/lib/pathfind.mjs` this round (none in engine/`src/` — every one of these was a bot-navigation-logic defect, not a gameplay-affecting bug):

1. **Stale straight-line waypoint resume after a loot detour** (level 5, all profiles) — re-BFS when drifted off a leg's expected path.
2. **Critical-health retreat freezing when cornered** — added a dedicated stall-strafe fallback for this branch (it skips the shared combat-stall bookkeeping via its own early return).
3. **`bfsPath` treating already-opened doors as permanently blocked** — added an `openDoors` set threaded through re-plan calls.
4. **`faceAngle` idling forever instead of turning** — it delegated all turning to `tick()`, which has no "just face this angle" branch.
5. **`angleDelta`'s atan2 branch-cut instability near an exact 180° turn** — pinned the turn direction near pi instead of trusting a potentially-flip-flopping sign (real, kept, though not the root cause of the freeze it was found chasing).
6. **Door-approach reach-check quantization gap** (the big one, 3 fix attempts) — replaced the fixed-step door push with a fine-stepped one so the bot can converge close enough to trigger the door regardless of collision-vs-reach-check granularity mismatches.

Net effect, worst-case stall length across a full 3-profile `npm run balancing:scan`: **500-600 ticks (multiple, systemic, several-per-scan) → 42-87 ticks (occasional, bounded, consistent with genuine short waits)**, verified stable across the final two consecutive full scans.

One residual, accepted, not fixed: the critical-health branch's stall-strafe fallback still can't always fully escape one specific cornering geometry — health can drain to 0 over a handful of 16-tick windows while the bot visibly (if not always successfully) repositions each window. Documented consistently since milestone 15 as a genuine difficulty/positioning signal, not a bug.
