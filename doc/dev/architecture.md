# Architecture

Codeenstein 3D is built as four strictly-layered stages, each of which only knows about the plain-data contract the previous stage hands it:

```
fs  →  parser  →  map  →  engine
```

For the actual file listing, see the root [`README.md`](../../README.md)'s "Project Structure" section. This document covers the contracts and rules between layers, and why they exist — for *why a given rule was chosen over an alternative*, see [Design Decisions](decisions.md).

## `fs` — source acquisition

`src/fs/workspace.ts` (local, via the File System Access API), `src/fs/github.ts` (a public GitHub repo's tree + lazily-fetched raw file content), and `src/fs/demoCampaign.ts` (the bundled `demo-campaign/` showcase campaign, inlined into the app's own bundle at build time via `import.meta.glob` — no picker, no network) all produce the same `TreeNode` shape. Nothing downstream — file tree UI, entrypoint detection, level launching, replay — ever special-cases which one supplied a given file; a `TreeNode`'s `handle` is a union wide enough to cover a real `FileSystemFileHandle` and a minimal `RemoteFileHandle` (which the demo loader also uses, wrapping an already-in-memory string instead of a network fetch).

## `parser` — source → normalized AST data

`src/parser/types.ts` defines `ParsedFile` and states the rule directly in its own doc comment: *"This module defines the ONLY shape the rest of the engine is allowed to know about... Nothing outside `src/parser/` should import `web-tree-sitter`."* A `ParsedFile` is plain, serializable JSON: `language`, `linesOfCode`, `entities` (functions/methods/classes/globals, each with a `complexityScore` and `nestingDepth`), `gotos` (resolved `goto`/label pairs), `comments` (large comment blocks), and `secretTriggers` (dead code, empty catch blocks, deprecation markers, commented-out code, and magic-number/blob literals — source for secret rooms).

Dispatch is by file extension (`src/parser/registry.ts`), against a single shared Tree-sitter WASM bootstrap (`src/parser/runtime.ts`, `Parser.init()` memoized so it only runs once regardless of how many adapters need it). PHP and C get bespoke, hand-written adapters — their grammars have quirks (PHP's global-variable-at-program-scope detection, C's buried function declarator) a generic traversal can't capture precisely. The other 12 bundled languages share one data-driven `GenericParserAdapter` (`src/parser/generic/`), driven by a shared node-type vocabulary rather than 12 separate hand-written adapters — see [Dependency Minimalism](decisions.md#dependency-minimalism).

`src/parser/security.ts` sits ahead of every adapter: an extension whitelist (only extensions a registered adapter claims ever reach a parser), a 4 MiB size cap, and a binary-content sniff (non-printable byte ratio over a sample window). Its own doc comment is explicit about scope: *"Nothing in this file — or anywhere else in `src/parser/` — ever evaluates, compiles, or executes the text it inspects. Source text is only ever handed to `Parser.parse()` ... or plain string/regex scans."* A parse failure logs a warning and skips the file rather than crashing the map generator or game loop.

## `map` — AST data → level data

`src/map/mapGenerator.ts`'s `MapGenerator.generate(parsed: ParsedFile, ...)` turns a `ParsedFile` into a `GameMap` (`src/map/types.ts`): a tile grid plus `Enemy[]`, `Room[]`, hazards, doors, keys, teleporters, traps, and pickups. Like `ParsedFile`, a `GameMap` is plain, serializable data — nothing in it references the engine, the player, or the DOM. Generation is deterministic: a seed is derived from a hash of the parsed AST itself (`seedFrom()`), fed through the shared `mulberry32` PRNG (`src/prng.ts`), so the same source file always yields the same map.

`mapGenerator.ts` itself is only the orchestrator: every placement subsystem is a pure function module under `src/map/generation/` (`corridors`, `breakup`, `doorsKeys`, `enemies`, `props`, `pickups`, `lore`, `secretRooms`, `teleporters`, `trapsHazards`, `spawnExit`, `labyrinth`, `geometry`, `pathing`, `seed`, `util`), each taking the shared grid, accumulator arrays, and the seeded `rng` explicitly. `generate()` is the single place their ordering — and therefore the deterministic rng draw sequence — is defined; reordering calls there changes every existing map layout and desyncs every recorded replay.

One post-carve pass doesn't derive from the AST at all: `breakUpLongCorridors` scans the fully-carved grid for straight corridor runs past a length threshold and interrupts each one with a small injected room (or a forced jog if the room won't fit), tracked separately as `GameMap.breakupRooms: Rect[]` rather than folded into `Room[]` — it has no backing `CodeEntity`, so every AST-driven system stays untouched by construction rather than needing a guard clause. Its own "Edge Case" enemies (`Enemy.edgeCase`) are the one enemy type not derived from a parsed entity either — see [Corridor Breakup: Best-Effort, Not Guaranteed](decisions.md#corridor-breakup-best-effort-not-guaranteed). Every room being reachable from spawn, by contrast, *is* a hard invariant (`assertAllRoomsReachable` checks it at the end of every `generate()` call) — see [Room Connectivity Is a Hard Guarantee, Corridor Length Isn't](decisions.md#room-connectivity-is-a-hard-guarantee-corridor-length-isnt).

### "Enemy stays data"

`Enemy` (and similarly `Mine`, other trap types) is a bare interface — position, HP, cooldown timers, a `home` room rectangle, `aggroed`/`discovered` flags, a back-reference to the `CodeEntity` it represents — with **zero methods**. The doc comment on `Enemy.attackCooldown` states this directly: *"Behaviour lives in `src/engine/enemyAi.ts` — this stays plain data."* All actual behavior (roam/chase/melee/ranged state machine, line-of-sight, steering) lives in free functions in `src/engine/enemyAi.ts` that take the `Enemy[]` array, the `Player`, and the `GameMap`, and mutate the plain data in place. `enemyAi.ts`'s own header explains why: *"This lives in the engine layer rather than as a method on the `Enemy` data ... so the map never depends on the player."* The same split applies to traps (`src/engine/traps.ts` mutates plain `Mine`/`SpikeTrap` data on `GameMap`).

Practical effect: the map layer can be generated, seeded, and reasoned about — including in the replay system — with no dependency on player state, input, or rendering at all.

### Why `difficulty.ts` and `prng.ts` live at `src/` root

Both `src/difficulty.ts` and `src/prng.ts` are needed by more than one layer, and the map layer must never import the engine layer (the reverse is fine and expected). Rather than duplicate either module or force an awkward import direction, both live at `src/` root, outside `map/` and `engine/` entirely — layer-neutral by construction. In practice, difficulty scaling is applied by `engine.ts` (enemy HP is rescaled in-place on the map's `Enemy` objects right after construction) rather than threaded through `MapGenerator.generate()`, but the module itself stays usable from either layer.

## `engine` — level data → running game

`src/engine/` (27 modules) consumes a `GameMap` + a `Player` and runs the render/simulation loop, reporting `EngineStats` back to `main.ts` via `EngineHandlers` callbacks. Key modules: `engine.ts` (game loop), `raycaster.ts` (renderer), `textures.ts` (wall/door/floor `TextureSet`: procedural defaults, or WAD-sourced via `src/wad/`), `player.ts`, `enemyAi.ts`, `pathField.ts` (one shared player-rooted BFS distance field all chasing enemies steer by, refloods only on player tile change or a grid mutation), `spatialGrid.ts` (tile-bucketed enemy index for rocket proximity/blast queries, rebuilt only on frames with rockets in flight), `sprites.ts`, `weapons.ts`, `ammo.ts` (the four ammo pools as one keyed record + per-pool metadata), `loot.ts` (drop rolls), `lootApply.ts` (drop/pickup application behind the narrow `LootContext`), `projectiles.ts`/`rockets.ts`, `traps.ts`, `scoring.ts`, `highscores.ts`/`defaultHighscore.ts`, `replay.ts`, `audio.ts`/`bgm.ts`, `hud.ts`, `viewmodel.ts`, `automap.ts`, `effects.ts`, `input.ts`, `storageCompression.ts`.

### Rendering

Pure Canvas 2D — no WebGL, no 3D library. `raycaster.ts` casts one DDA (Digital Differential Analyzer) ray per screen column at a 640×400 internal resolution (CSS-scaled up for a chunky retro look), drawing one shaded vertical wall strip per column. Shading is distance fog plus a fixed y-side dimming for fake directional lighting. Walls, doors, the floor, lore-terminal walls, and hazard/teleporter/spike-trap floor tiles are all texture-mapped: `src/engine/textures.ts` owns a `TextureSet` (one bitmap per slot — regular wall, bonus-level wall, door, floor, bonus-level floor, lore-terminal wall, hazard floor, teleporter floor, spike-trap safe floor, spike-trap active floor), generated by default as procedural patterns into detached offscreen canvases (a brick wall, a steel bonus-wall panel, a two-leaf door, a flagstone floor, and matching tone-appropriate panel/flagstone variants for the other five slots) — keeping the "no external asset files" approach the rest of the engine already uses (`audio.ts`, `viewmodel.ts`) — or, if the player loads a DOOM WAD file via the sidebar, real composited wall/door/lore-terminal textures and floor/hazard/teleporter/spike-trap flats parsed out of it by `src/wad/` (a from-scratch WAD/PLAYPAL/TEXTUREx/patch/flat binary parser — deliberately hand-rolled rather than an npm dependency, see [Dependency Minimalism](decisions.md#dependency-minimalism)), auto-selected per slot from a hardcoded name allowlist with a silent per-slot fallback to the procedural default if nothing matches. The renderer samples per-column via `drawImage` plus a shading overlay for walls/doors/lore-terminals, and per-pixel during the floor-cast for the floor/hazard/teleporter/spike-trap tiles. The ceiling is the only thing left deliberately *not* textured, staying flat-colored. Secret walls reuse the plain wall texture plus a subtle identifying tint overlay rather than getting their own slot (they must stay near-visually-identical to a plain wall); lore-terminal walls get their own real texture plus a much more visible pulsing tint overlay on top, since — unlike secret walls — they want to be noticed as interactive, and that pulse used to be their entire "this is a terminal" signal before texturing. Spike traps swap between two different WAD flats (a plain/metal one and a blood/lava one) depending on whether `activeSpikeTileKeys` says the trap is in its safe or damaging phase, rather than tinting one texture — the real signal a player needs, modeled as which flat is showing instead of a color. Hazard and teleporter floor tiles sample their real WAD flat with no tint at all, matching how Doom itself renders these; the JS-computed sine-wave pulse they used to have before texturing is gone (real Doom animates these via multi-frame flat-cycling, which this engine doesn't implement). Each column's wall top/bottom edge is kept as a float rather than floored to a whole pixel, and the one boundary row at each edge is alpha-blended (sampling the exact texel color for textured columns) against whatever's already painted underneath in proportion to its sub-pixel coverage — antialiasing the top/bottom silhouette against ceiling/floor without touching the `zBuffer` (still one exact distance per column, so billboard occlusion is unaffected). Vertical seams between adjacent columns (e.g. at corners) are not antialiased — that would need supersampling across columns, which is a bigger cost/architecture change (`notes` Task 71).

`src/wad/` is its own small acquisition layer, structurally analogous to `parser/`: plain byte-array-in/plain-data-out, with zero DOM/Canvas dependency (fully testable in plain Node — `scripts/verify-wad-parser.mjs`), and consumed only by `src/engine/textures.ts` — nothing else in the engine, and nothing in `map/`, ever needs to know the WAD binary format. It sits outside the `fs → parser → map → engine` pipeline proper: it's user-triggered at any time via the sidebar (not part of a level's fs/parser/map derivation), session-only (not persisted, matching the existing "Select BGM Folder" precedent), and never affects simulation/replay determinism — texture choice is purely cosmetic.

Enemies and items are 2D billboards (`sprites.ts`), projected with the same camera transform as walls. Every world billboard — enemies, projectiles, keys, ammo drops, the exit marker, teleporters, mines — is collected into one list and drawn in a single furthest-to-nearest depth-sorted pass against a per-column `zBuffer` recorded during the wall pass, so occlusion (by walls or by other billboards) is always correct regardless of draw-call order. This replaced an earlier fixed-order-by-category approach that could draw a nearer item behind a farther one — see [Feature Scope Reversals](decisions.md#feature-scope-reversals).

### Determinism and replay

Anything that affects simulation outcome — enemy AI roam/fire-cooldown timing, loot rolls, weapon cone-of-fire spread, elite-loot coinflips — draws from the shared seeded `mulberry32` PRNG (`src/prng.ts`), never `Math.random()`. Purely cosmetic randomness (blood-particle scatter, SFX pitch, BGM shuffle order, console hint selection) stays on `Math.random()` deliberately, since it never feeds back into simulation state. This split is a hard constraint, not a style preference: the replay system (`replay.ts`) records a gameplay seed plus a per-frame input digest per level, and can only reconstruct a byte-identical run if every simulation-relevant random draw goes through the one deterministic stream. See [Determinism & Replay](decisions.md#determinism--replay) for how this evolved.

## Build

Vanilla TypeScript + Vite 6, strict `tsc`, no game framework. `vite.config.ts` aliases `fs/promises`/`module` to `src/empty-node-shim.ts`, an empty stub — purely to satisfy Rollup's static resolution of `web-tree-sitter`'s bundle, which references a Node-only code path that's dead in a real browser but still needs to resolve at build time.

Unit coverage is a Vitest suite (`src/**/*.test.ts`, co-located next to what they test) with a 100%-lines/statements/functions/branches gate (`npm run coverage`), wired as a blocking CI job alongside the browser-driven scripts below — see [Testing](testing.md) for the full setup, shared mock catalog, and mocking philosophy; broad gameplay/visual e2e coverage beyond the scripts here remains an open backlog item in `notes`. There are four `scripts/verify-*.mjs` scripts in total, split across two CI jobs by whether they need a real browser: `scripts/verify-wad-parser.mjs` and `scripts/verify-demo-campaign.mjs` (see `notes` Task 76) are pure Node — the latter bundles the real parser + `MapGenerator` via esbuild (no browser needed) and gates on a full map-feature/enemy-type coverage checklist — and run in the `verify` (no-browser) CI job; `scripts/verify-wad-textures.mjs` and `scripts/verify-campaign-playthrough.mjs` are genuinely Playwright-driven, each spinning up a real headless browser against a real dev server, and run in the separate `verify-browser` job — as a `chromium`/`firefox` matrix (see [Testing](testing.md#cross-browser-verification) for why both scripts are safe to run against a non-Chromium engine). `verify-campaign-playthrough.mjs` drives the real app end-to-end (via an OPFS-stubbed workspace picker and a virtual clock) to verify the save/highscore/replay mechanism; `verify-wad-textures.mjs` loads a synthetic in-memory WAD and confirms wall/door/floor/lore-terminal/hazard/teleporter/spike-trap texturing actually reaches the live renderer, saving a screenshot for manual spot-checking of anything not worth a hard pixel assertion. Both `scripts/generate-default-highscore.mjs` and `scripts/run-balancing-telemetry.mjs` drive the game through the same shared `Bot` class (`scripts/lib/bot.mjs`) — real navigation (`scripts/lib/routePlanner.mjs`, on top of `pathfind.mjs`'s BFS) and combat (turns toward/fires at nearby aggroed enemies via the undocumented `Backquote` automation-only fire key — see `isFireHeld()`'s doc comment in `src/engine/input.ts` — no mouse/Pointer Lock needed), each configured via explicit constructor options rather than reading `process.env` itself. `scripts/lib/qualifyLoop.mjs` holds the retry-until-N-qualifying-runs loop both scripts use; `scripts/lib/virtualClock.mjs` holds the virtual-clock installer both share.

`scripts/generate-default-highscore.mjs` (`npm run generate:default-highscore`, `notes`) is a one-shot dev tool, not a verifier: for each bot skill profile (Casual/Gamer/Pro), it plays `demo-campaign/` until 3 runs qualify (reach campaign level 4/5/6 respectively), keeps the single highest-scoring qualifying run per profile, and bakes the resulting 3 entries into `src/engine/defaultHighscore.ts` — the shipped fallback leaderboard shown to a first-time player with an empty highscore board. The file is a single gzip+base64-encoded string constant (`gz1:` prefix, same scheme `storageCompression.ts` uses for `localStorage`), not a plain array literal — a qualifying run's replay can carry tens of thousands of frames, and both the shipped bundle size and the cost of a bundler parsing the data mattered enough to compress it (an early uncompressed version of this file reached ~84MB). `loadHighscoresForDisplay` (`highscores.ts`) decompresses it on demand via `decompressFromStorage`. Rerun the generator (and review the diff) whenever `demo-campaign/`'s source files change, or its baked-in replay `astHash` values go stale and playback refuses.

`scripts/run-balancing-telemetry.mjs` (`npm run balancing:telemetry`, `notes` "balancing") is a third headless-Chromium bot script, built for data collection rather than shipping an artifact: it plays `demo-campaign/` with three bot profiles (Casual/Gamer/Pro, `scripts/run-balancing-telemetry.mjs`'s `PROFILES`) across all three difficulties, retrying each of the 9 profile×difficulty combos until it collects 3 "qualifying" runs (a run qualifies once it clears level 4 — earlier deaths are discarded outright), and writes a self-describing 7-category telemetry breakdown (map density/enemy demographics, combat pacing/TTK, AI effectiveness, damage/healing, weapon efficiency, economy/loot, navigation) per level and per campaign to `balancing_telemetry.json` at the repo root (gitignored — regenerate on demand, not a build artifact). All runtime data comes through dedicated additions to `window.__codeensteinTestHooks` — `getMines()` and `getTelemetrySnapshot()` — backed by new pure tracking types/helpers in `src/engine/telemetry.ts` and `RaycasterEngine`'s existing `?testHooks=1`-gated instrumentation (zero overhead when the flag is off); nothing here uses a fresh `Math.random()` call — bot decisions stay deterministic via the engine's existing seeded `mulberry32` streams (note: the top-level *gameplay seed* itself is still randomized fresh per process launch, so two separate invocations of the same script at identical settings are not expected to produce byte-identical output). Each combo's attempts run concurrently (`ATTEMPT_CONCURRENCY`, default 12, `CODEENSTEIN_TELEMETRY_CONCURRENCY` env override) via isolated Playwright browser contexts sharing one Chromium process; a single flaky context crashing (`page.evaluate: Target page, context or browser has been closed`, observed intermittently under concurrency) is caught per-attempt and discarded rather than taking down the whole run, with a circuit breaker if the browser itself appears to have died. A full 9-combo run is a long, mostly-unattended job (tens of minutes to a few hours depending on how many attempts each combo needs) — see `CODEENSTEIN_TELEMETRY_HEADED=1` for watching one attempt play in a real (non-headless) browser at human-readable speed when diagnosing bot behavior, and `CODEENSTEIN_TELEMETRY_LEVEL_LIMIT`/`CODEENSTEIN_TELEMETRY_ATTEMPT_CAP`/`CODEENSTEIN_TELEMETRY_PROFILE`/`CODEENSTEIN_TELEMETRY_DIFFICULTY` for scoping a smoke test down. See [Balancing Telemetry Bot](balancing-telemetry.md) for the full operational reference — every env var, the permanent `npm run balancing:scan` regression detector, and a critical headed-vs-headless timing gotcha (a convergence check tight enough to work perfectly under the virtual clock can be structurally unreachable in a real, visibly-rendered browser).
