# Game Design

This is the developer-facing "why" behind Codeenstein 3D's design — for the player-facing "how", see [`doc/user/mechanics.md`](../user/mechanics.md); for exact current numbers (HP multipliers, aggro radii, drop rates), see that doc or the source directly, since they change as balancing continues.

## Everything maps to something

The core design principle is that every construct the parser can extract from a source file should become *something* in the level, rather than being silently dropped — a codebase should feel fully translated, not partially. Concretely:

- **Functions/methods → enemies.** HP scales with cyclomatic complexity, so a genuinely tangled function is a harder fight than a simple one — the game's difficulty curve is, by design, a reflection of the codebase's own complexity rather than a hand-tuned level progression. A function with its own code smells (more than 5 params, or nesting more than 3 levels deep) gets a scaled HP bonus on top of the base complexity number — the smell itself, not just the branching that produced it, is treated as something worth fighting harder.
- **Global variables → hazard rooms**, not enemies. Globals aren't "doing" anything the way a function is; they represent ambient, code-smell-adjacent state, which reads better as an environmental hazard (an acid pool) than as something you fight and defeat.
- **Private/protected methods → key-locked rooms.** Encapsulation becomes a literal locked door — access modifiers already express "you're not supposed to reach this directly," which maps naturally onto needing a key.
- **`goto`/label pairs → teleporters**, in the spirit of Unreal Tournament jump pads: a `goto` is already a non-linear jump in control flow, so a bidirectional teleporter pair is a fairly direct translation rather than a stretch.
- **Code smells that hide something → secret rooms.** Originally just dead code (unreachable statements after a `return`) — narratively, code that can never execute is a hidden space the codebase itself doesn't know is there, fitting for a secret behind a fake wall with better loot. Dead code alone turned out to be too rare a trigger in clean, well-tested codebases, so the pattern was broadened to four more "the code is hiding something" smells: an empty/swallowed-exception catch block, a `@deprecated`/`[Obsolete]`-style marker, a commented-out code block (or an oversized comment block generally), and a magic-number/blob literal (a long Base64-ish string, or a well-known hex constant like `0xDEADBEEF`). All five feed the same capped, fairly-sampled, one-per-room pool of secret-room slots per level (see `placeSecretRooms` in `mapGenerator.ts`), and the level-start briefing now surfaces the resulting count so a player knows there's something to look for. Loot is picked uniformly among whatever's actually available that run — health and armor are always in the running, rockets once the launcher's owned, and a still-unowned gdb/ghidra/Friday Hotfix outright (plus, from campaign level 4 on, Toolchain) — rather than the original two-way health/rockets coinflip, which collapsed to "always health" for any player who hadn't unlocked the launcher yet.
- **Large comments → lore terminals.** A big comment is the author speaking directly to a future reader; surfacing it as an interactable in-world text panel keeps that voice intact instead of discarding it. TODO/FIXME comments are deliberately exempted from the normal length gate (even a one-line `// TODO: fix this` gets a terminal) because that's the most common real-world form of technical debt, and excluding short ones would miss most of what this feature exists to surface — each one also spawns a small "technical debt" encounter (trap, mine, or a weak "Bug" enemy) on the approach, rather than a permanent hazard, since the terminal has to be reachable to read — kept at least a minimum distance from the player's spawn point, same as every other hazard placement, so a TODO comment that happens to sit in the spawn room can't ambush the player before they can react (see [Hazard Placement Spawn Safety](decisions.md#hazard-placement-spawn-safety)). License/copyright headers are excluded on purpose — they're boilerplate, not authored lore, and would otherwise turn nearly every file into a terminal.
- **`.h` files → bonus levels.** A header is usually a smaller, self-contained, single-purpose file compared to its implementation — a natural fit for a shorter "restock" level with boosted pickup rates, discovered "by accident" during playtesting rather than planned from the start.

## Enemy and difficulty philosophy

A function's cyclomatic complexity converts to HP at a fixed rate; beyond a certain size, complexity is expressed as *more* enemies (a "pack") rather than one increasingly bloated HP pool, so a big function reads as "this does a lot" rather than "this is a damage sponge." Past an extreme complexity threshold, that logic flips: instead of yet another incrementally bigger pack, the function spawns a single Elite — a real boss fight (higher HP multiplier, more damage, a guaranteed health/ammo drop plus a high-odds extra weapon drop) marking a genuinely different tier of code, not just "a really big pile of the same thing." See [Enemy Scaling](decisions.md#enemy-scaling-packs-vs-elites) for how this threshold was chosen.

Easy/Normal/Hard scale three axes in the same direction on purpose: Hard makes enemies tougher, deal more damage, *and* aim straighter, while simultaneously making ammo/health/armor scarcer — the effects compound rather than offset, so difficulty isn't just "bullet sponges," it's a genuinely tighter resource margin and a sharper threat too. Easy mirrors most of this in reverse, but not damage dealt one-for-one with its HP reduction: balance telemetry (a 450-run, 9-combo campaign) found Easy's original mirrored curve, combined with a cautious/health-seeking playstyle, could produce near-total immunity for an entire 17-level campaign — real stakes on Easy needed the damage floor raised independently of the HP floor, so `DIFFICULTY_MULTIPLIERS.easy.damage` (0.85) no longer mirrors `.hp` (0.7) the way Hard's pair still does.

Enemy aim is the third axis, added after that same telemetry found accuracy sitting in a narrow ~70-77% band regardless of difficulty tier — enemies had *zero* aim deviation at all, so difficulty's whole bite was HP/damage, never actually "smarter." Ranged bolts now roll a random deviation off dead-center, up to a per-difficulty cap (`enemyAimSpreadDeg`: 10°/4°/0° for easy/normal/hard) — Hard's 0° keeps enemies aiming exactly as precisely as they always did (a bolt still isn't hitscan or homing, so it stays dodgeable), while Easy/Normal enemies now genuinely aim worse rather than just hitting proportionally softer.

Every other enemy exists *because* of something the parser found (a function, a TODO comment); the Edge Case is the one exception — it's a pacing fix, not a code-mapping one. Long, empty corridors read as boring "endless walk" filler no matter how well the rest of the level maps to the source, so a straight corridor run past a length threshold gets a small breakup room injected into it (or, if the room won't fit, a forced jog around a short severed stretch instead), and that room is populated with 1-3 Edge Cases — small, fast, jarringly cyan-tinted, barely-any-HP nuisances that read as a literal "bug in the system" rather than a real threat. The geometric half (shorter corridors) and the gameplay half (something to shoot on the way through) are deliberately both present rather than either alone, since neither one fully solves "the walk feels empty" by itself.

## Weapon and economy intent

Ammo is split into four pools (bullets, smg, rockets, gas) specifically so each weapon has a distinct resource identity rather than one shared pool making every gun interchangeable. gdb originally shared the bullets pool with the pistol/shotgun — split out into its own `"smg"` pool once that stopped reading as a distinct resource identity in practice: a full-auto weapon burning through a shared pool at ~11 rounds/sec meant a single pickup sized for a precise pistol shot (6 bullets) drained in under a second, and diluted the "each weapon should feel distinct" intent this section already committed to. Its own pool gets its own drop size (`SMG_DROP_AMOUNT`, a magazine-sized 30) instead. Friday Hotfix (the flamethrower, unlocked at campaign level 12) followed suit from day one with its own `"gas"` pool (`GAS_DROP_AMOUNT`, also 30) rather than repeating gdb's original mistake. gdb, ghidra, and Friday Hotfix are all gated behind an unlock (an Elite kill, or a forced grant at fixed campaign milestones as a safety net) — giving any of them out from the start would make an Elite's guaranteed drop meaningless, and rolling their ammo before the weapon is owned would just be a dead, wasted drop. Toolchain (the chainsaw that replaces the knife once found) deliberately breaks that pattern rather than extending it: it has no forced-unlock milestone at all, and isn't reachable through the ordinary per-kill roll either — only a secret room, an Elite kill, or a small independent chance riding on an otherwise-empty regular-kill roll (see below) — all from campaign level 4 on — a genuinely missable unlock, not a "you'll definitely have it by level N" guarantee like the other three. Where gdb (full-auto hitscan) and ghidra (splash-damage projectile) each cover a distinct combat role, Friday Hotfix instead leans into being a point-blank melter that's nearly useless at range, giving the campaign's latest unlock a genuinely different identity from "gdb but stronger." Cone spread alone didn't sell that short range in playtesting — pellets could still land well past a real flamethrower's reach even with a *narrower* cone than the shotgun's — so it also carries a hard `maxRange` (3.5 tiles), the only weapon with one; every other ranged weapon leaves range entirely to the Cone of Fire's spread/falloff.

Loot rolls exclude outcomes that would be pointless in context: smg/rocket/gas ammo never drop before the matching weapon is unlocked, so a drop the player can't use at all doesn't happen — it would read as the game wasting their luck. Health used to be part of the same weighted roll (skipped only at full stability); it's now its own always-on check instead, independent of everything else — any regular kill tops the player up if they're not at full stability, no roll involved. That split happened after balance telemetry found cutting health the same way as ammo (see below) collapsed Hard's clearing rate: health is the one resource that directly ends a run when it runs short, unlike ammo, which stays survivable via the knife/Toolchain melee fallback. Ammo/armor still come from one weighted roll per kill, but that roll doesn't always land anymore — a 20% chance (`REGULAR_KILL_NO_DROP_CHANCE`) it drops nothing at all, added after the same telemetry found dynamic (kill-drop) supply so overwhelmingly dominated static level placement (93-100% of everything actually consumed, every resource type) that ammo never felt meaningfully scarce regardless of level design.

The SIGKILL Knife (melee, unlimited ammo, small lifesteal) exists as a deliberate safety net: since a kill's ammo/armor roll can now come up empty (and always could, before a weapon is unlocked, or once a pool runs dry between drops), the knife's job is to make sure a player who's fully dry can still fight their way back into the loop rather than getting stuck. Toolchain takes over that same job once found — a stronger chainsaw with double the damage and a bigger heal-per-kill, permanently replacing the knife on Space rather than sitting alongside it, and (unlike the knife) firing continuously while held instead of once per press. It doesn't remove the safety net, it just upgrades it for whoever gets lucky enough to find it. Toolchain itself gained a third way to get lucky: alongside the pre-existing secret-room and Elite-bonus-roll paths, a regular kill's ammo/armor roll coming up empty now carries its own small independent chance (5%) to grant Toolchain instead of nothing at all — added after balance telemetry found the other two paths were, in practice, close to unreachable (a shortest-path bot never explores for secret rooms at all, and an early clean Elite kill is rare).

## Scoring intent

Scoring is built to reward the behaviors the game actually wants to encourage, not just "kill things": points scale with the complexity of what was defeated (tripled for an Elite), plus bonuses for finishing with health/ammo to spare, for speed, and for route efficiency (actual path walked vs. the shortest possible spawn→exit path) — and a separate bonus for reading lore terminals, for opening secret rooms (double the lore bonus per room, since finding one takes actively interacting with a specific stretch of wall rather than just walking near a terminal), and for exploring most of the map, so pure speedrunning isn't the only path to a high score. Chaining kills fast also pays off directly: 3 kills within 3 seconds of each other triggers a "MULTI KILL!" bonus, and 6 within 6 seconds an "ULTRA KILL!" (bigger bonus, matching the top-tier map-completion bonus), each with its own on-screen banner and procedural stinger — a rolling-window check, so a long streak only announces once per tier crossed rather than spamming every subsequent kill. Score is a running **campaign total**, not a per-level number that resets — a multi-level run is meant to read as one continuous performance, not a sequence of disconnected level scores. (This was originally broken — see [Persistence & Storage Limits](decisions.md#persistence--storage-limits) — and fixed once identified.)

## Known design gaps

A few open items in [`notes`](../../notes)' `## Open` section are design-facing rather than pure engineering backlog, and are called out here so this doc doesn't read as though the design is finished:

- **Balancing** is explicitly still open — numbers throughout (HP curves, drop rates, aggro distances) are expected to keep moving.
- **Room decorations** were implemented once, disabled twice after playtest feedback (felt like visual clutter rather than atmosphere), and are sitting behind a flag pending a design rethink — not deleted, because the underlying idea (rooms feeling less empty) is still worth pursuing.

See `notes` directly for current status on any of these — this doc intentionally doesn't track checkbox state.
